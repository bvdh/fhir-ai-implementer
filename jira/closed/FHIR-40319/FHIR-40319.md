# FHIR-40319: Simplify comparison definitions

## Metadata

- **__EMPTY**: Elliot Silver
- **__EMPTY_1**: Public
- **__EMPTY_2**: As an example of the confusion the current language creates, consider:




eq
the range of the search value fully contains the range of the target value


lt
the range below the search value intersects (i.e. overlaps) with the range of the target value


le
the range below the search value intersects (i.e. overlaps) with the range of the target value or the range of the search value fully contains the range of the target value





I interpret `eq` as:

	`search.start &lt;= target.start &lt;= target.end &lt;= search.end`



I interpret `lt` as "there is some portion of target that is below the search value", which turns into:

	`target.start &lt; search.start`



Now, lets try `le`. The definition of `le` is the union of `lt` and `eq`, which suggests:

	`(target.start &lt; search.start) or (search.start &lt;= target.start &lt;= target.end &lt;= search.end)`



However, that leads a strange result where if `(target.start &lt; search.start)` is true, we don't care about `target.end` and it can be less than, *greater than *or equal to `search.end` , but if `(target.start &lt; search.start)` is false, then `target.end` becomes significant and it cannot be greater than `search.end`, which is not a natural interpretation of `le`. 

Given the definition of `lt`, I would expect `le` to be define as:

	`target.start &lt; search.end`



Maybe my expectations are off, maybe my interpretations are off, or maybe the spec is off, but given the language of the spec, I can't tell. Simplifying the language would clarify it for all.

- **Description**: Current text:
the range above the search value intersects (i.e. overlaps) with the range of the target value
Can we simplify the language for these comparisons to something like:
The upper boundary of the target value is greater than the upper boundary of the search value.
Or using the proposed language from FHIR-40029:
the upper boundary of the parameter value.
Turning this into simple comparisons using the upper and lower bounds of the parameter and resource values would greatly simplify the mental effort for anyone trying to convert these into code. Right now, they need to draw (at least mental) diagrams of the range of the search value and the area above it, and figure where the range of the target value can be in relation.

That way we can easily see that for eq it is saying
The lower boundary of the resource value is greater than or equal to the lower boundary of the parameter value, and the upper boundary of the parameter value is less than or equal to the upper boundary of the resource value.
Additionally diagrams would assist in understanding.

### 2. Elliot Silver - 2/23/23
Security: Public

Reviewing the table, this is my interpretation, of what it wants to be saying (not necessarily what the current text implies, especially for le and ge on ranges). Current text is included for reference.
If param is the parameter value (for range values, converted to a precise range), and res is the value in the resource (for range values, converted to a precise range), then:



eq 
the value for the parameter in the resource is equal to the provided value
res.exists() and res = param
the range of the search value fully contains the range of the target value
res.exists() and param.start &lt; res.start and res.end &lt; param.end


ne
the value for the parameter in the resource exists and is not equal to the provided value
res.exists() and (res = param)
the range of the search value does not fully contain the range of the target value
res.exists() and !(param.start &lt; res.start and res.end &lt; param.end)


gt
the value for the parameter in the resource exists and is greater than the provided value
res.exists() and res &gt; param
the range above the search value intersects (i.e. overlaps) with the range of the target value
res.exists() and param.end &lt; res.end


lt
the value for the parameter in the resource exists and is less than the provided value
res.exists() and res &lt; param
the range below the search value intersects (i.e. overlaps) with the range of the target value
res.exists() and res.start &lt; param.start


ge
the value for the parameter in the resource exists and is greater or equal to the provided value
res.exists() and res &gt;= param
the range above the search value intersects (i.e. overlaps) with the range of the target value, or the range of the search value fully contains the range of the target value
res.exists() and param.start &lt; res.end


le
the value for the parameter in the resource exists and is less or equal to the provided value
res.exists() and res &lt;= param
the range below the search value intersects (i.e. overlaps) with the range of the target value or the range of the search value fully contains the range of the target value
res.exists() and res.start &lt; param.end


sa
the value for the parameter in the resource exists and starts after the provided value
res.exists() and res &gt; param
the range of the search value does not overlap with the range of the target value, and the range above the search value contains the range of the target value
res.exists() and param.end &lt; res.start


eb
the value for the parameter in the resource exists and ends before the provided value
res.exists() and res &lt; param
the range of the search value does not overlap with the range of the target value, and the range below the search value contains the range of the target value
res.exists() and res.end &lt; param.start


ap
the value for the parameter in the resource exists and is approximately the same to the provided value. Note that the recommended value for the approximation is 10% of the stated value (or for a date, 10% of the gap between now and the date), but systems may choose other values where appropriate
res.exists() and res ~ param
the range of the search value overlaps
res.exists() and (res.start &lt;= param.end or param.start &lt;= res.end)




Note that "(or for a date, 10% of the gap between now and the date)" should be removed since date comparisons are handled as ranges.
