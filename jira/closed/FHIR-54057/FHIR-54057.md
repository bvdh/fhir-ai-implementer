# FHIR-54057: Clean up conditional* elements in CapabilityStatement

- **Created**: 1/13/26
- **Description**: There are now several 0..1 fields in CapabilityStatement related to conditional interactions

(conditionalUpdate, conditionalCreate, conditionalPatch)

All of these should be redundant now that these interactions are able to be represented in the "interaction" element.

Note: conditionalRead does not appear to fall under this case - since that's a code, but conditionalDelete does. (Since the delete-conditional-single and delete-conditional-multiple codes exist on interaction, evidence of one of those renders the conditional-delete code a bit redundant)

- **Key**: FHIR-54057

- **Related URL**: http://build.fhir.org/capabilitystatement.html
- **Reporter**: HaydenBader
- **Resolution**: Persuasive with Modification
- **Resolution Description**: We will add read-conditional-modified-since and read-conditional-not-match codes

Will mark conditionalCreate, conditionalRead, conditionalUpdate, conditionalDelete, conditionalPatch as deprecated with instructions to use the new interaction codes.  Will also deprecate the associated value sets and code systems (where they exist)
