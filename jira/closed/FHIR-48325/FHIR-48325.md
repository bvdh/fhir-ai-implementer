# FHIR-48325: Binding strength shouldn't be required when the value set reference isn't present

## Metadata

- **Assignee**: Unassigned
- **Created**: 9/19/24
- **Description**: ElementDefinition.binding.strength should not be required if binding.valueSet is not populated, as having a binding "strength" without any value set that the element is actually being bound to is entirely meaningless (and almost certainly may be confusing).  Changing this would most likely involve relaxing the binding.strength cardinality constraint from 1..1 to 0..1 and adding an invariant that requires binding.strength to be present if binding.valueSet is populated.  This would allow bindings to contain only a description, supporting the cases where a general idea of what should be used to populate the element can be described, but there is no actual value set available or able to be created at that time to support the binding statement (see Jira discussion https://chat.fhir.org/#narrow/stream/179165-committers/topic/Description.20for.20coded.20elements.20without.20binding).

Since ElementDefinition is normative, I'm not certain that this change can be made, but since this would be a (slight) loosening of the current constraint, it may be possible?  An alternative could be requiring the binding.valueSet canonical reference to always be present (i.e. updating the binding.valueSet cardinality to 1..1), but that most likely would be too constrained and is a breaking change, so that is likely neither desirable or possible.
- **Issue Type**: Change Request
- **Key**: FHIR-48325
- **Related Page(s)**: Datatypes
- **Related URL**: https://build.fhir.org/elementdefinition.html#ElementDefinition
- **Reporter**: Robert Hausam
- **Resolution**: Persuasive with Modification
- **Resolution Description**: We will add an additional binding strength of "descriptive" with a definition of "There is no constraint on the allowed set of codes at the root level, though there may be additional bindings that apply in certain contexts.  The root binding element will be a description that gives guidance on potential downstream constraint of the allowed codes and/or highlights the presence of additional bindings."

We will add an invariant that says "if strength = 'descriptive', valueSet must be omitted".

We will update the terminology page that discusses bindings to explain the use of this new strength.
- **Status**: Applied
- **Summary**: Binding strength shouldn't be required when the value set reference isn't present
