- **Description**: We should provide more guidance about expectations for validation results.  Specifically, the need to differentiate between "issues with the instance" and "issues with the validation process".  Implementers need to understand when they see a result whether it means "this instance is definitely broken" as opposed to "there might be an issue here that the validator can't electronically check" vs. "the validation process outright failed and even issues that would normally be caught haven't been checked."

In particular, we should be clear about the meaning of error vs. warning vs. information when we're talking about validation.

My opinion:

Error - An issue that shows that the instance is definitely invalid.  (Not met cardinality, invariants, vocab validation)
Warning - An issue that shows that the instance might be invalid but the validator can't tell for sure (non-met extensible binding, declared profile the validator can't find, declared code outside a binding where the system is unknown, etc.)
Information - An issue that doesn't impact the validity of the issue, but may still indicate something that could be a problem or could be better, or that provides guidance to help understand other errors or warnings in the instance (e.g. elements in an instance that aren't expected to be supported, violated best practice recommendations (if the validator hasn't been set to treat those as errors), elements that didn't match any slices during open slicing, slicing information, etc.)

For conformance testing, an error indicates a definite failure.  A warning indicates a potential failure.  Information messages can be ignored, but they're useful to authors of examples or when diagnosing systems.

### 3. Grahame Grieve - 9/26/24
Security: Public

Just in terms of what actually happens now - or at least what the general intent is:

	Error is used for unexpected process failures inside the validator (server didn't respond, so similar), as well as for things that are known not to be comformant
	In addition to when conformance cannot be known (code systems unavailable, or extensible bindings), warning is also used for SHOULD rules, and for a number of different security and implementation considerations, and invariants labelled 'warning', and various best-practice rules



But whatever definitions we come up with, there are a number of parameters that implementers can provide to the implementation process for various reasons including legacy arguments about what should or shouldn't be an error, and any definitions should be clear that implementers still have operational freedom in this regard
