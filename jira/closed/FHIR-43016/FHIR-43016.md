# FHIR-43016: Allow implicitRules to repeat

- **Created**: 10/26/23
- **Description**: Not sure if we can get away with this at this point, given that Resource is normative.  However it would certainly be useful.

The issue is if a given instance has two sets of special rules that it wants implementations to adhere to when dealing with the instance.  For example, let's say it wants to require support for the new &#91;rendering-CriticalElement&#93;(https://jira.hl7.org/browse/FHIR-42990) extension and also wants all currencies to be automatically assumed to be EUR.  There are standard URIs for both of these 'rules'.  And you might have a client system that recognizes both of those URIs.  However, there's no way to send both at once right now - so the only solution is for an IG author to define a new URI in their particular IG and say "this URI means 'follow the rendering-CriticalElement' rule as well as the 'everything's a Euro' rule" - which is a URI that might well not be recognized by lots of systems that recognize the other two.
- **Issue Type**: Change Request
- **Key**: FHIR-43016
- **Related Artifact(s)**: BaseResource
- **Reporter**: Lloyd McKenzie
- **Resolution**: Persuasive with Modification
- **Resolution Description**: We will define a new modifier extension called additionalImplicitRules 0..* of type uri with definition "Additional implicit rules beyond the single policy that can be pointed to by Resource.implicitRules."

comment: "There is no ordering implied within the list of multiple rules, nor prioritization between which rule appears as 'implicitRules' and which appear in this extension.  It is possible to use this extension even if no implicitRules element is present (e.g. because the author doesn't want to pick one of the rules to be sent elsewhere)"

We will update the Resource.implicitRules to add the comment:

"If more than one implicitRules policy applies, additional policies may be conveyed using the [additionalImplicitRules] extension."
