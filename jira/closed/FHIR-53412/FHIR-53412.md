- **Description**: The Timing datatype is used in the Dosage datatype, where it constitutes an important part of the Dosage functionality. We've been doing last minute analysis on the Dosage data type, focusing on complex dosage schedules that contain multiple different dosages for different times and contexts - pre/post treatment, tapering dosages, etc. 

As part of this, we've identified 4 elements that should be added to timing to support these complex interactive/interlaced schedules:



startOffset 0..1 SimpleQuantity:

definition: Indicates that the events within the period of the repeat do not start until startOffset has elapsed

comments: This element is defined to support interlaced schedules, e.g. for different actions that happen on different days, but that have repeat cycles that are tied together by the context. This is a usually encountered in Dosages

invariant: (error) only present when frequency is present and &gt; 1. 

---- 

endOffset 0..1 SimpleQuantity:

definition: Indicates that the events within the period of the repeat step endOffset before the period ends

comments: This element is defined to medication schedules that cycle like 'do something for X days, and then stop for Y days' 

invariants:

  (error) only present when frequency is present and &gt; 1

  (warning) endOffset and endEvent - event might terminate without warning 

---- 



startEvent 0..1 RelativeTime:

definition: Indicates that repeating periods do not start until the time specified by the relative time.

comments: This is to support medications that start post medication, or a tapering dose regime that starts once the medication is successful

---- 

endEvent 0..1 RelativeTime:

definition: Indicates that repeating periods stop at the time specified by the relative time.

comments: This element is defined to medication schedules that stop medication once an event happens (take this medication until X)

invariants:

   (warning) endOffset and endEvent - event might terminate without warning 

---- 

Note that these events are not the same as the existing event property, because that even property relates to when the repeating events happen, whereas these last two event related elements control the periods themselves.

It's possible that we could make these elements extensions, and consider elevating them to elements later, but that might be less than ideal because of the modifier question.

As part of this, we should provide a more algorithmic description of how the different properties of Timing.repeat combine; there's some lack of clarity there for implementers. (we could do that anyway, but it would be difficult to document that well without turning these elements as extensions into modifier extensions which would be problematic)
