- **Description**: &gt; Server refers to the FHIR application or applications acting as the subscription provider. This includes the responsibilities of implementing one or more

The FHIR application is not a client, it has to be a FHIR Server otherwise the mechanisms do not work. Suggest to replace with:

&gt; Server refers to the FHIR server acting as the subscription provider. This includes the responsibilities of implementing one or more

(an completely out of band server that just sends notifications and nothing else might be the exception, if this is a use case we support, indicate this exception in the out-of-band section as an exception)
- **Grouping**: [AwaitingMerge]

- **Related Section(s)**: 3.9.1
- **Reporter**: Bas van den Heuvel
- **Resolution**: Not Persuasive with Modification
- **Resolution Description**: Implementations of Subscriptions are not required to be part of the same application or service that acts as a FHIR Server.

In order to clarify, will update the spec to:

> *Server* refers to the FHIR application or applications acting as the subscription provider, such as a FHIR Server or monitoring system converting internal events into FHIR notifications.

Note that the distinction is to avoid implications that an application providing subscription functionality exposes things like a FHIR REST API, etc.. There are architectures and use cases where they are disconnected.
