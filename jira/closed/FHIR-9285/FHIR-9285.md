# FHIR-9285: StructureDefinition/ElementenDefinition binding should be 0..* not 0..1

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/15/16
- **Description**: In V3 there is no rule against binding an element against multiple value sets. I've asked Rob Hausam/McClure this and it is also not rules out in the newest binding project. In a significant number of our current templates in NL we bind to more than one value set, and StrucDoc does the same (checked with Austin).

         In FHIR however you can only bind to one valueset with the apparent assumption that you will use value set imports to get to other value set. Unfortunately this hinders a smooth transition from CDA (V3) to FHIR.

         Hence I'd like to propose to make the cardinality of the binding element 0..* instead of the current 0..1. Note that multiple bindings will be needed more for the logical models type of StructureDefinition than for implementation, but in case it, they are needed.
