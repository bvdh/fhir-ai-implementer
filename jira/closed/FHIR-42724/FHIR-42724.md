# FHIR-42724: Remove Requirements.satisfiedBy

## Metadata

- **Assignee**: Unassigned
- **Created**: 9/11/23
- **Description**: Requirements should have not references to the artifacts which satisfy this requirement.

The workflow is that there is Requirements first, then there are artifacts designed to satisfy this requirement.

So, It should be in a reversed manner such that the external artifact has reference to the requirement.

- **Resolution**: Not Persuasive with Modification
- **Resolution Description**: We will:
 # Add to this sentence: "This is for use when the requirement is met be a pre-existing artifact e.g. a rule that's met by the base FHIR spec, or a national implementation guide." the following - "It is more typical to point from the FHIR artifact satisfying the requirement to the requirement (using the [satisfies-requirement]([https://build.fhir.org/ig/HL7/fhir-extensions/StructureDefinition-satisfies-requirement.html)] extension) to the Requirements resource.  However, in some cases the author of the Requirements and satisfying resource(s) won't be coordinated in that manner and this element allows assertion of satisfaction without involvement from those managing the implementing artifacts.
