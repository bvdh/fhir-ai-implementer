- **Description**: The error and recovery scenarios in section 3.9.10.4 briefly mention that subscriptions can be resumed after an error state. All example diagrams have eventsSinceSubscriptionStart (and eventNumbers) resume from the last count when the client recovers/when notifications are being sent again, but that behavior is fully defined anywhere in the narrative.

 

Should eventsSinceSubscriptionsStart (and eventNumbers) reset to 0 when a Subscription goes from and errored state back to requested/active?

 

(I'm unsure what the right answer is, it likely has implications on how many "failures" the server will tolerate before updating a Subscription status to errored - i.e. temporary connectivity issues shouldn't reset eventsSinceSubscriptionsStart, but multiday outages might)
