- **Description**: We're evaluating a change to how our FHIR server builds the self link in search response bundles, specifically when access control restrictions apply filters that correspond to existing, known search parameters.

Currently, the self link includes only the parameters explicitly supplied by the client. For example:


GET /Observation?subject=Patient/123

Produces:


"self": "https://example.org/fhir/Observation?subject=Patient%2F123"

However, if access control (e.g., scopes) silently limits results to a specific category—say, category=laboratory should the self link reflect that constraint?

Proposed:


"self": "https://example.org/fhir/Observation?subject=Patient%2F123&amp;category=laboratory"

This would only apply when the internally applied filter maps to a standard search parameter.

According to the FHIR spec (3.2.1.9) Server Conformance:
In order to allow the client to be confident about what search parameters were used as criteria by the server, the server SHALL return the parameters that were actually used to process the search.
What's unclear is whether this applies to filters added implicitly by the server for access control purposes, especially when those filters correspond to defined query parameters.

We're considering this update to improve transparency and align with spec intent without exposing internal logic unrelated to supported search parameters.

Should the section 3.2.1.9 account for this ?
