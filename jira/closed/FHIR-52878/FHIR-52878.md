- **Description**: The current regex for the dateTime data type is the following:

```
(&#91;0-9&#93;(&#91;0-9&#93;(&#91;0-9&#93;&#91;1-9&#93;|&#91;1-9&#93;0)|&#91;1-9&#93;00)|&#91;1-9&#93;000)((0&#91;1-9&#93;|1&#91;0-2&#93;)((0&#91;1-9&#93;|&#91;1-2&#93;&#91;0-9&#93;|3&#91;0-1&#93;)(T(&#91;01&#93;&#91;0-9&#93;|2&#91;0-3&#93;):&#91;0-5&#93;&#91;0-9&#93;&#91;0-5&#93;&#91;0-9&#93;|60)(\.&#91;0-9&#93;
{1,9})?)?)?(Z|(&#43;|-)((0&#91;0-9&#93;|1&#91;0-3&#93;):&#91;0-5&#93;&#91;0-9&#93;|14:00)?)?)?
```

This currently allows certain partial dates (more specifically YYYY-MM, and YYYY-MM-DD) to have the time zone component present. For example: `2025-09Z`, and `2025-09-16+01:00` are both valid. But it doesn't allow time zone with a year-only partial date. For example `2025+00:00` is not valid.

According to the language itself, and as Grahame clarified on https://chat.fhir.org/#narrow/channel/179166-implementers/topic/Timezone.20optionality/near/539609742, the intented behaviour is for the dateTime format to be a subset of ISO8601.

Whilst the regex does not have to enforce all rules, it would be great for it to be less lenient when it comes to invalid dateTime strings.

Proposed change:

```
(&#91;0-9&#93;(&#91;0-9&#93;(&#91;0-9&#93;&#91;1-9&#93;|&#91;1-9&#93;0)|&#91;1-9&#93;00)|&#91;1-9&#93;000)((0&#91;1-9&#93;|1&#91;0-2&#93;)((0&#91;1-9&#93;|&#91;1-2&#93;&#91;0-9&#93;|3&#91;0-1&#93;)(T(&#91;01&#93;&#91;0-9&#93;|2&#91;0-3&#93;):&#91;0-5&#93;&#91;0-9&#93;&#91;0-5&#93;&#91;0-9&#93;|60)(\.&#91;0-9&#93;{1,9}
)?(Z|(&#43;|-)((0&#91;0-9&#93;|1&#91;0-3&#93;):&#91;0-5&#93;&#91;0-9&#93;|14:00)?)?)?)?)?
```
