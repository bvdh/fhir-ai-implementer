# FHIR-57354: Remove restriction on Parameters in Bundle.entry.resource requiring reference from another entry

## Metadata

- **Assignee**: Unassigned
- **Created**: 5/19/26
- **Description**: Bundle.entry.resource currently states: "This is allowed to be a Parameters resource if and only if it is referenced by something else within the Bundle that provides context/meaning."

This restriction should be removed. Two problems:

	It is written as prose but stated as a hard rule ("if and only if"). If we meant it, it should be a constraint/invariant. It is not currently enforceable.
	There are legitimate use cases for Parameters entries that are not referenced by another resource in the Bundle. Examples:
	
		Subscription notifications on topic types whose notification payloads are modeled as Parameters.
		Responses to batch/transaction interactions.
		Other contexts where a Bundle legitimately carries one or more Parameters entries as primary content.
	
	



Background: this guardrail was added under FHIR-26390 alongside the broader FHIR-26368 work that loosened Parameters' top-level definition. The intent at the time was to constrain where Parameters could appear after relaxing the general prohibition. Our understanding of Parameters' utility has since evolved, and the specific Bundle.entry.resource restriction no longer reflects current usage or carries clear value.

Proposed resolution: Remove the sentence from Bundle.entry.resource. Do not replace it with a constraint.
