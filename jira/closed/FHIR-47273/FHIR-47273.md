# FHIR-47273: Async-bundle: removal of results

- **Description**: The async-bundle pattern does not state when the data returned in the result is no longer available. Is the data automatically deleted once the Status Request returns an Error/Complete or can a a client depend on the data to be available after such response? 

I suggest to:
1. update the delete section to state that calling delete on a completed request will remove any status data.
2. state in 3.2.6.2.4 that the client should not assume the data to be present after receiving a Response-Error or Response-Complete result.
- **Issue Type**: Change Request
- **Key**: FHIR-47273
- **Related Page(s)**: Asynchronous API (async)
- **Related Section(s)**: 3.2.6.2.4.0.3 / 3.2.6.2.3
- **Reporter**: Bas van den Heuvel
- **Resolution**: Persuasive with Modification
- **Resolution Description**: Will change "Following the delete request, ..."

to

"Following the delete request, the server SHOULD delete all data associated with the request and..."

Will add an STU note saying:

"Servers MAY delete the data associated with a request after some period of time has passed, either since the data became available or after a status request that returned the data.  (Immediate deletion after returning the data is not recommended as it is possible client may not receive the response.)  Implementer feedback on recommended timeframes for data retention are welcome.  For now, policies will vary by server.  Clients which wish to ensure the data is deleted after retrieval can use the Delete process."
