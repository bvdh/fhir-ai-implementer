# FHIR-48702: Pipe symbol (as version delimiter) needs to be percent-encoded in HTTP GET request

## Metadata

- **__EMPTY**: David Booth
- **__EMPTY_1**: Public
- **__EMPTY_2**: Based on the zulip discussion at https://chat.fhir.org/#narrow/stream/179177-conformance/topic/syntax.20of.20Reference.2Ereference.20with.20version/near/373036174 , and particularly on Grahame's comment, it looks like the correction for these HTTP GET examples is not to percent-encode the pipe symbol, but to put the version in a query string.  For example, this incorrect example:


GET [base]/QuestionnaireResponse?questionnaire:below=http://example.org/fhir/questionnaire/patient-intake|1 

should apparently be:


GET [base]/QuestionnaireResponse?questionnaire:below=http://example.org/fhir/questionnaire/patient-intake?version=1

However, I notice that there are a lot of incorrect examples of HTTP GET requests on the page, showing a pipe symbol in the URI of the GET request, which is not syntactically allowed per RFC 3986.  Many of these examples have the option of being displayed as either as HTTP GET requests or HTTP POST requests, and in the POST version, the syntactic constraints are different, so a pipe character would be syntactically allowed in that context.  This leads me to wonder whether the FHIR community's intent really is to shift the version into a query parameter only in the HTTP GET case.
- **Assignee**: Unassigned
- **Comments**: 3/11/25
- **Created**: 10/24/24
- **Description**: As explained in https://build.fhir.org/search.html#versions , the pipe symbol (vertical bar) is used to separate the version from the canonical URI.   However, when this is placed in the query string of a GET request, the pipe symbol must be percent-encoded, because a bare pipe symbol is not syntactically permitted in a query string: https://www.rfc-editor.org/rfc/rfc3986#page-49 

 

Currently the GET example with a version is: 

GET &#91;base&#93;/QuestionnaireResponse?questionnaire:below=http://example.org/fhir/questionnaire/patient-intake|1

It should be:

GET &#91;base&#93;/QuestionnaireResponse?questionnaire:below=http://example.org/fhir/questionnaire/patient-intake%7C1

 

See also zulip discussion at https://chat.fhir.org/#narrow/stream/179177-conformance/topic/syntax.20of.20Reference.2Ereference.20with.20version/near/373036174

However, I believe Eric Prud'hommeaux's suggested correction in that thread unnecessarily percent-encodes the colon after "http".  RFC 3986 allows colon characters in a query string.
