# FHIR-48379: Add warning that FHIRPath expressions might break if the maximum cardinality changes

- **Created**: 9/24/24
- **Description**: https://jira.hl7.org/browse/FHIR-47643 raised the issue that JSON parsing might break if someone changes the maximum cardinality of a field.

During discussion, it was noted that this also affects FHIRPath, and resources that use FHIRPath, because suddenly you get either multiple resources when expecting 1 or 1 resource when expecting multiple.  There should be some warning that FHIRPath expressions need to be reviewed when moving to a new version of FHIR.
- **Grouping**: [Applied]
- **Issue Type**: Change Request
- **Key**: FHIR-48379
- **Related Page(s)**: FHIRPath (fluentpath)
- **Related URL**: https://build.fhir.org/fhirpath.html
- **Reporter**: Paul Lynch
- **Resolution**: Persuasive
- **Resolution Description**: 1. We will change "Note that this may change the path to the element in some syntaxes (e.g. JSON)."

to "Note that this may change the path to the element in some syntaxes (e.g. JSON) and may also break FHIRPath expressions that don't allow for the possibility of repetitions."

2. Will change "Because of the JSON issue, changes to cardinality..." to "Because of the JSON and FHIRPath issues, changes to cardinality..."

3. Will add a dragon warning on the FHIR FHIRPath page that says "FHIR inter-version compatibility rules allow for non-repeating elements to be made repeating in future FHIR versions.  For this reason, FHIRPath expressions SHOULD be written presuming all elements will repeat, even if the current version of the model has a maximum cardinality of '1'."

4. Will ask @brian Postlethwaite to check our current invariants and SearchParameter definitions to identify any FHIRPaths that are not defined in a way that's safe if maximum cardinalities change.
- **Status**: Applied
