# FHIR-48421: Basic should define a pattern for custom search parameters

## Metadata

- **__EMPTY**: Josh Mandel
- **__EMPTY_1**: Public
- **__EMPTY_2**: Discussion from FHIR-I

&gt; There should be guidance to assure that these custom search parameters are clearly distinct to that custom resource definitiion.

If my Basic has "code" of 
{"system": "foo", "code": "bar"}

SP might be have an indexing expression like:
Basic.where(code.exists(system = "foo", code = "bar")).extension('http://my-identifier-property')

... but the question here becomes: what url-param-name do we give it to avoid conflicts?

We could write up guidance for this, e.g. incorporating the system + code into the short name. like "foo-bar-identifier"

- **Description**: Basic resource is intended for custom resource definition. As such there is a need for the profiler of basic for their custom resource to define search parameters for their custom resource. There should be guidance to assure that these custom search parameters are clearly distinct to that custom resource definitiion.
