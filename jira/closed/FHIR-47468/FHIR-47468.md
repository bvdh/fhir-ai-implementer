- **Description**: Suggest adding parameters to control the file size of NDJSON of the Response - Complete.

A client may not want the NDJSON of the same type of FHIR resource generated by the FHIR server to be too small or too large. For a large volume of particular FHIR resources, small file size of NDJSON lead to too many small files that a client needs to request remotely. Too large single NDJSON may fail very often when downloading and also difficult to process for some resource limited client. Therefore, minimaNDJSONFileSize and maximaNDJSONFileSize can be specified as part of the request parameters.

- **Resolution Description**: ### Additional Query Parameters

| Query Parameter | Optionality for Server | Optionality for Client | Cardinality | Type | Description |
|----------------|------------------------|------------------------|-------------|------|-------------|
| _minimumFileSize | optional | optional | 0..1 | number | Specifies the minimum size in bytes for generated NDJSON files. The value SHALL be a positive integer. If a server supports this parameter, it SHOULD construct files that meet or exceed this size, unless doing so would violate the _maximumFileSize constraint.  |
| _maximumFileSize | optional | optional | 0..1 | number | Specifies the maximum size in bytes for generated NDJSON files. The value SHALL be a positive integer and SHALL be greater than _minimumFileSize if both are specified. If a server supports this parameter, it SHALL construct files that do not exceed this size. The server MAY use a lower internal maximum.  |

### Implementation Notes

1. If neither parameter is specified, servers use their default file size behavior.
2. Servers MAY deviate from specified sizes when necessary (e.g., last file in sequence).
3. Servers SHOULD document any fixed bounds they implement.
