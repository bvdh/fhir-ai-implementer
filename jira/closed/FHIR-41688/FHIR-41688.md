# FHIR-41688: Define search parameter for the Range datatype

## Metadata

- **__EMPTY**: Gino Canessa
- **__EMPTY_1**: Public
- **__EMPTY_2**: I see a few paths forward:

	describe range searches in terms of composite search parameters (if allowed),
	define new search types for range data,
	add a new modifier, or
	a general new 'and' value delimiter



Below I am going through each option in a bit more detail.  I am trying to cover as much as possible, but these are not comprehensive.

If general consensus is composite:

	Add content to the search page in Composite Search Parameters, with links from data types that could have 'range' mechanics.
	Add examples showing definitions and requests for each of the relevant data types.
	Add a request for WGs to review resource-defined search parameters to see if they should include range definitions as well.
	Example: `date-range=gt2023-04-01$le2023-05-1`, `quantity-range=ge10|mmol/L$lt20|mmol/L`.
	Questions
	
		Can you define a composite search parameter that repeats a single SP? (required for this to be a valid option)
	
	
	Pros
	
		No additional structural definitions in FHIR
		Easy to use in earlier versions of FHIR
	
	
	Cons
	
		WGs and IG authors need to 'double define' search parameters
		Users need to change search parameters if they are searching for 'simple' ranges (e.g., 2023) vs. 'complex' ranges (e.g., 2023-04-01 to 2023-05-01).
	
	



If general consensus is new search types:

	Define a new search type for each range.  Input format needs to support two values, with optional units on the types that need them:
	
		lower-value|&#91;lower-unit&#93;|upper-value|&#91;upper-unit&#93;
		Values are allowed to include prefixes
	
	
	Results are: date-range, number-range, and quantity-range
	Add the search types to search types code system
	Add sections with documentation and examples defining each type in Basic Parameter Types (Search Parameters).
	Add a request for WGs to review resource-defined search parameters to see if they should include range definitions as well.
	Example: `date-range=gt2023-04-01|le2023-05-1`, `quantity-range=ge10|mmol/L|lt20|mmol/L`.
	Pros
	
		Very flexible within each type.
	
	
	Cons
	
		WGs and IG authors need to 'double define' search parameters
		Users need to change search parameters if they are searching for 'simple' ranges (e.g., 2023) vs. 'complex' ranges (e.g., 2023-04-01 to 2023-05-01), with different semantics.
		Challenging to use in earlier versions of FHIR (can define `special` search parameters that use the same behavior)
	
	



If general consensus is a new modifier:

	Determine separation character (using `$` in examples here for sanity)
	Define a single modifier `:range` on compatible types.
	Use prefixes to determine inclusivity/exclusivity of boundaries
	Example: `date:range=gt2023-04-01$le2023-05-1`, `quantity:range=ge10|mmol/L$lt20|mmol/L`.
	Pros
	
		Allows reuse of existing search parameters
	
	
	Cons
	
		Use in earlier versions of FHIR means adding an unknown modifier to that implementation
		Difficult/impossible to use with composite search parameters.
	
	



If general consensus is a new delimiter:

	Determine the delimiter character that will be used for joining multiple values as a single component and search
	Similar to the current 'or' semantics, different from current 'and' joins in that they are applied together (e.g., both have to apply to the same element value repetition)
	Example: `date=gt2023-04-01*le2023-05-1`, `quantity=ge10|mmol/L*lt20|mmol/L`
	Pros
	
		Allows reuse of existing search parameters
	
	
	Cons
	
		Use in earlier versions of FHIR means adding a new character delimiter to that implementation
