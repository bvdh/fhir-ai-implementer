# FHIR-16198: Add delete/purge REST operation

## Metadata

- **__EMPTY**: Lloyd McKenzie
- **__EMPTY_1**: Public
- **__EMPTY_2**: What is the difference between the proposed operation and the existing DELETE operation?  FHIR allows a server to execute a delete as a logical or physical delete.  From the perspective of the querying system, the results are the same.  Are you wanting something that would remove/hide all historical records of the resource as well?  (Even if you did, the server would need to prevent creating new records with the old id.)
- **Assignee**: Gino Canessa
- **Comments**: 5/3/18
- **Created**: 5/3/18
- **Description**: In some cases, the resources need to be permanently purged from a FHIR server. Currently early adopters lack this mechanism and it's critically importnat to move a system from test into production.  Testing a system will create incomplete/invalid data that needs to be purged. 

         The purge operation could create an audit event. The purged data could be recorvered from a db backup.
- **Grouping**: [AwaitingMerge, ReOpen]
- **Issue Type**: Change Request
- **Key**: FHIR-16198
- **Related Page(s)**: REST (http)
- **Related URL**: http://hl7.org/fhir/2018May/http.html
- **Reporter**: Ioana Singureanu
- **Resolution**: Persuasive with Modification
- **Resolution Description**: We do not believe this is only for test data management.  Functionality like this is widely used in production.  We will define the following capabilities:

From Q6 discussion at WGM: Plan is to define:

DELETE /:type/:id/_history - remove all versions of the resource except the current version (which if the resource has been deleted, will be an empty placeholder)
DELETE /:type/:id/_history/:vid - remove the specified version of the resource.  It is an error to remove the 'current' version.  (Must first perform a regular delete, and can then delete the non-current version.  If the desire is to roll-back to a previous version, use a PUT to "roll forward" instead.)

Same status codes as for the original DELETE operation

Will define the `delete-history` and `delete-history-version` to correspond to these

POST /Patient/:id/$purge – get rid of all current + historical data for a whole Patient compartment
POST /Group/:id/$purge – get rid of all current + historical data for all Patient compartments of all group members

These $purge operations are suitable for async behavior.

Will include a link to both of these operations on the http page in the Delete section.

### 3. Ioana Singureanu - 5/29/18
Security: Public

A "deleted" can be brought back and it's permanently in the database. A"purged" database will go away for good.

         If "purged" resouce is not "production" it "real" and it can be removed from the database without a tracker. During testing and training users, a application will produce a lot of garbage data that could be purged.

         "purge" like "lock/unlock" woud be admnistrative operations, not applicable to clinical apps but part of the testing and clinical data.

         I'm raising this issue in ballot because I foresee a lot variability in the future for FHIR server side implementers using custom operations to purge test data.  If the solution is "create your operation" some guidance in the standard spec would be useful such that these custom operations are consistent across FHIR server implementations.

### 11. James Agnew - 1/21/19
Security: Public

Ewout asked me to join the fun and comment here about the motivation for HAPI FHIR's $expunge operation.

         For what it's worth, our original use case was pretty mundane stuff: It was added to support the SMART project, whose reference server is intended to work like a publically accessible "normal" FHIR server, including deletes and the ability to see those deletes. They wanted a complete reset overnight every night though, including any data submitted by the public being complely physically removed at that time. A custom operation seemed like a logical solution, and making this a standard feature in HAPI seemed reasonable if we were going to do the work anyway.

         It's probably also worth noting that this feature is disabled by default in HAPI FHIR. I don't know how many people use it but it's come up twice on our mailing list since the feature was added so at least some people find it useful.

         I personally find it useful in that I actually end up invoking this operation fairly regularly in various contexts, although I have yet to ever use it on a production system for anything "real".
