# FHIR-41183: R5 ElementDefinition.id metadata specifies id data type instead of string

- **Description**: FHIR R5 metadata StructureDefinition-ElementDefinition.json appears to be wrong. We ingest it the same way we ingest R4 and STU3 metadata, but now with R5 our internal FHIR validator is rejecting numerous R5 example resources because it thinks that R5 ElementDefinition.id should adhere to data type "id", not "string".

The HL7 FHIR spec web site definitely gives you the impression that ElementDefinition.id - by way of Element.id - is string.

However, in each metadata snippet shown here, see the "base" definition. R4 has "valueUrl":"string" and R5 has "valueUrl":"id".
