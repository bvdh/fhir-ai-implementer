# FHIR-50108: Bundle.entry.response.etag format is unclear

The quotes are required by the HTTP standard: https://httpwg.org/specs/rfc9110.html#field.etag
- **Assignee**: Unassigned
- **Comments**: 3/25/25
- **Created**: 3/24/25
- **Description**: While implementing a bundle transaction/batch processor I discovered that documentation for the content of the Bundle.entry.response.etag format is unclear.

When used as a header this property is intended to be W/"1" so this is what I put into my code for the entry.response.etag, then compared it to the spec examples and HAPI which are both different...

The documentation for what the ETag value should be is here:

http://hl7.org/fhir/http.html#versioning

And the examples in the spec also have something different too: W/1 (no quotes - with escaping)

http://hl7.org/fhir/bundle-response-medsallergies.json.html

Adding to the confusion, there is a contradictory statement in the definition of the etag property

https://build.fhir.org/bundle-definitions.html#Bundle.entry.response.etag

"Etags match the Resource.meta.versionId. The ETag has to match the version id in the header if a resource is included"

Which of the 2 matches here is intended to be explicit? the header is formatted with the content of the versionID, so they both can't match. Suggest at a minimum changing one of the "match" words to consistent with.

The HAPI engine just uses 1 (no quotes or W/) in the entry.response.etag property.
