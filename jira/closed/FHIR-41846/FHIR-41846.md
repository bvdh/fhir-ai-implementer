# FHIR-41846: Rename ElementDefinition.requirements

## Metadata

- **Assignee**: Unassigned
- **Created**: 8/21/23
- **Description**: The description of the intended meaning of the ElementDefinition.requirements field states

"Why this resource has been created"

which suggests that this "requirements" field has more so the semantics of a "reason" or "purpose".

 

Indeed the analogous field in the StructureDefinition resource, with semantics "Why this structure definition is defined", is called "purpose"

 

Unless there is still valid historical context I'm missing for why ElementDefinition.requirements should be named so; I recommend changing it to "purpose" both for clarity and consistency with aforementioned similar concepts in other resources.
