# FHIR-33172: The $lookup operation states that some parameters have a choice of type which is not supported by OperationDefinition

## Metadata

- **__EMPTY**: Carmela A. Couderc
- **__EMPTY_1**: Public
- **__EMPTY_2**: Discussed at Q2 9/23/2021 WG meeting - joint Vocab and FHIR-I

A normative resource uses non-normative extensions. According to Lloyd, this is acceptable.

To date, extensions typically are not normative. However, they could be. Maturity levels of extensions have not been tracked. 

An alternative is to add the extension to the OperationDefinition resource. 

Is the extension supported by 80% (ish) of the implementer community, or is it widely in use? 

Another possibility is to make an explicit reference to extensions on the base resource page (minimize backward compatibility issues)

Because OperationDefinition is normative and changing cardinality is a breaking change - should be done with caution. The extension may have been introduced to avoid changing cardinality. 

OR - the extension was introduced as an ad-hoc measure to satisfy publishing requirements, and the creator didn't have time to submit a change request to OperationDefinition. There is nothing about the extension definition that means it could not become part of the base definition.

Another wrinkle - the $lookup operation is normative, and it uses the extension. If the extension is moved, would there be unintended impacts?

GG: mentioned that APIs would not have to change, but code generators would.

Why is this of concern to the vocabulary community?

Vocab is addressing several tickets related to property - concern was raised about the use of a non-normative extension for a normative resource.

The core spec doesn't address the ability for a parameter (input or output) to have multiple types.

Reuben will add a ticket to FHIR-I to add the extension to core (multiple types for parameters). This ticket is based on a mis-understanding. Reuben will withdraw this ticket.
- **Assignee**: Unassigned
- **Comments**: 9/23/21
- **Created**: 8/13/21
- **Description**: The $lookup operation page states that the

	 property.value and 
	property.subproperty.value



parameters each support of choice of the following types:

	Coding | boolean | code | dateTime | decimal | integer | string



However, the OperationDefinition resource only allows the type of a parameter to have a single type.

Noting that both the $lookup operation and the OperationDefinition resource are normative, it is suggested that this issue be resolved jointly by Vocabulary WG and FHIR-Infrastructure work groups.
