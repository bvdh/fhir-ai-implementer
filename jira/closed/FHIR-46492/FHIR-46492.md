# FHIR-46492: Additional Granularity in Notifications

## Metadata

- **__EMPTY**: Gino Canessa
- **__EMPTY_1**: Public
- **__EMPTY_2**: The core issue this ticket aims to address is that currently:

	subscriptions are requested at topic granularity
	
		filters and shapes are defined/instantiated at the resource type granularity
	
	
	notifications are issued per subscription - which by definition is topic granularity



Topic-based subscriptions are concerned with triggers and notifications, using a subscription as the linking mechanism.

As the work was done defining these mechanisms, a lot of flexibility was left in by design - especially regarding pieces we did not have a lot of implementation experience with. In this document, we are specifically concerned with the granularity of triggers and subscriptions.

For details on the current entity relationship between topics, subscriptions, and notifications, see the attached diagram: &#91;^subscription-er-diagram.svg&#93; .

Specific considerations:

	Implementers have enough granularity to support triggers in ways that make sense to them.
	Subscribers can ask for notifications at any granularity listed above.
	Filters need to be applied consistently, e.g.:
	
		patient - e.g., notifications for me as a patient
		patient membership - e.g., notifications for a practitioner on a care team
		Arbitrary orgs/practitioners/etc. in the relevant resource - e.g., referral destination.
	
	
	Subscribers can determine the exact trigger for a notification.



To try and distill the issues above:

	SubscriptionTopic resources can define any number of triggers, whether or not they are related.
	Subscription resource map a single SubscriptionTopic to a single subscriber.
	Notifications are mapped to a single subscription.

- **Description**: Request to update R6 to match additional functionality requested in the Backport IG. (See related FHIR-43888).

In order to reduce the number of topics necessary to cover real-world use cases, some topics are being designed that cover multiple triggering events. However, right now notifications are only able to describe data down to the topic level.

Discussion at WGM included discussion on the R6 modeling.  Final result was to merge the event and resource triggers and add event tagging to notifications, based on the descriptions in the proposal document here.

- **Resolution Description**: Note: see comments for additional context.

{{SubscriptionTopic}} changes:
* *trigger*
** {{resourceTrigger}} and {{eventTrigger}} are collapsed into a single backbone element named {{trigger}}.
* *filter*
** Filters are _only_ defined per trigger.
*** {{canFilterBy}} is moved under {{trigger}}.
* *shape*
** Shapes are _only_ defined per trigger.
** {{notificationShape}} is moved under {{trigger}}.

{{Subscription}} changes:
* Subscriptions should/can tag filters by event.  If no event is specified, the filter should be applied to any trigger that contains a matching definition.
* *filter*
** Add {{event}} - {{CodeableConcept 0..*}} element to {{filterBy}}.

{{SubscriptionStatus}} changes:
* Notification instances can be explicitly linked to a specific {{trigger}} via the {{triggerEvent}}.  If there is no event defined by a trigger, the granularity stops at the level of the topic.
* Add an {{triggerEvent}} - {{CodeableConcept 0..*}} element to {{SubscriptionStatus.notificationEvent}}.

Include in all documentation that the event tagging is optional.  IGs are able to set additional requirements to add additional requirements.
