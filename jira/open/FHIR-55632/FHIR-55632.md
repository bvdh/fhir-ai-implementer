# FHIR-55632: Improve participant type pattern

## Metadata

- **Assignee**: Unassigned
- **Created**: 2/6/26
- **Description**: The ActivityDefinition resource defines the following elements:

ActivityDefinition.participant.type: 0..1, code
ActivityDefinition.participant.typeCanonical: 0..1 canonical
ActivityDefinition.participant.typeReference: 0..1 Reference

The names of these elements are problematic in that typeCanonical and typeReference appear to be choices of type, but they are not. Although not technically incorrect, this pattern can cause issues with implementations that make use of type name conventions as part of element processing, and the pattern is discouraged.

To better align with recommended methodology, we should:

1. Remove typeReference and typeCanonical (they were introduced in R5)
2. add detail&#91;x&#93; (canonical | Reference) (with the definition and constraints as specified in the typeCanonical and typeReference elements)

Note that this same pattern appears in:

PlanDefinition.actor.option.type
PlanDefinition.action.participant.type
RequestOrchestration.action.participant.type

We should apply the same updates to each of those locations.
