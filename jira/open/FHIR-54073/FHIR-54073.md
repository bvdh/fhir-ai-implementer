# FHIR-54073: Terminology functions should be fluent like most fhirpath functions

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/13/26
- **Description**: The set of %terminology factory functions are not like the rest of the fhirpath functions which are fluent in nature permitting easier usage. The existing memberOf(), subsumes() and subsumedBy() functions are already like this.

I'd like to suggest replacing the existing set of factory style functions with this new set of fluent functions (inspired if not directly copied from CSIRO's pathling extensions)

(also consider if any of these would require the "params" parameter that was included in the factory functions for some of the uncommon parameters - as noted in FHIR-54069)

 
useTerminologyServer(...)
which modifies the evaluation engine's current configuration with the value for anything chained after it. This would then also work for the subsumes function too. (Similar to how `defineVariable` works)
(this would/could replace the %terminology.at() function)
e.g. Patient.gender.useTerminologyServer('&lt;TSserverURL&gt;')
.translate('http://example.org/ConceptMap/local-gender-codes')
// translate the patient gender into a local coding based on a specific concept map on a specific terminology service
This could be set at the start of the expression then don't need to restate it anywhere else in the expression. Not required to be immediately before the termonologies function call.
 
display(&#91;language: String&#93;): collection&lt;string&gt;
get the display value for each input code/coding
 (one of possible uses of lookup - without all the extra result processing needed)
e.g. item.where(linkId='a1').answer.value.display('fr') // retrieve the French display text for the coded response
 
designation(use: coding, &#91;language: string&#93;): collection&lt;string&gt;
lookup/retrieve the designation for the input coding(s) - could use TS lookup function
(one of possible uses of lookup - without all the extra result processing needed)
 
property(code: string, &#91;language: string&#93;)
lookup/retrieve the property for the input coding(s) - could use TS lookup function, output type will be whatever the type of that property is defined.
Note: the weight() function could have partially been implemented using this function.
(one of possible uses of lookup - without all the extra result processing needed)
_e.g. {{item.where(linkId='a1').answer.value
.property('http://hl7.org/fhir/StructureDefinition/itemWeight')}}_
 
translate(conceptmapUrl, reverse, equivalence, targetSystem, &#91;sourceSystem&#93;): collection&lt;coding&gt;
works on input collections of type `code` (needs sourceSystem), `coding` or `codeableConcept`. Source and TargetSystems are canonical URLs that can also contain the version too.

e.g. Patient.gender.translate('http://example.org/ConceptMap/local-gender-codes') 
// translate the patient gender into a local coding based on a specific concept map
 
validateCode(system: url, &#91;date: dateTime&#93;, &#91;language: string&#93;, &#91;abstract: boolean&#93;)
canonical (optionally versioned), input collection code, coding, codeableconcept, returns OperationOutcome or boolean?
Could have variations for VS/CS i.e. validateCodeInCS/validateCodeInVS
 
expandValueSet()
simple function, same params as the factory proposal
Indicate why the factory approach is preferable, and include this justification/reasoning in the specification.
What other use cases does %terminologuy.lookup provide that the above simpler to use functions provide?
