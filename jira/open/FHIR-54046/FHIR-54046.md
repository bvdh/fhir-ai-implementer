# FHIR-54046: SubscriptionTopic.trigger.event should be 0..*

## Metadata

- **Assignee**: Gino Canessa
- **Created**: 1/13/26
- **Description**: It seems like SubscriptionTopic.trigger.event should be 0..*

While having multiple event triggers could be shown via repetitions of SubscriptionTopic.trigger, that would also require you to repeat everything else in the trigger model (canFilterBy, notificationShape, etc.)

The situation I have in mind is a SubscriptionTopic for a long group of ADT events (but possibly not all of them, i.e. no "parent" event to use). In that situation, there are multiple distinct events that all happen to act the same as a trigger (in terms of filters, shape, etc.). Expressing this via repetitions of SubscriptionTopic.trigger is possible, but would be very duplicative.
- **Issue Type**: Change Request
- **Key**: FHIR-54046
- **Related Artifact(s)**: SubscriptionTopic
- **Reporter**: James Jahns
- **Resolution**: Persuasive
- **Resolution Description**: Will update {{SubscriptionTopic.trigger.event}} to be {{0..*}}.  Will add documentation explaining that multiple {{CodeableConcept}} values should only be used when representing different events (e.g., consolidating the trigger for multiple patient-status-changed events) and clarify that using different codes representing the same event should still be placed in the same {{CodeableConcept}} value as multiple {{coding}} values (e.g., using an ADT event code and a local hospital code that represents the same vent).

Note that this is already available in the {{SubscriptionStatus}} resource, so no changes are required there.
