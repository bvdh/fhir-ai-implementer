# FHIR-53878: When would ETag be used on conditional-delete-single?

## Metadata

- **Assignee**: Gino Canessa
- **Created**: 1/9/26
- **Description**: The conditional-delete-single interaction allows a client to delete a resource/s based on some selection criteria rather than by a specific logical ID. It also defines an If-Match header containing a weak E-Tag to ensure that the resource to be deleted is up to date.

In most scenarios, if a client has access to the correct ETag value, they would also have access to the FHIR ID and could just do a regular Create request. I can't think off the top of my head of what counterexamples might be.
- **Issue Type**: Change Request
- **Key**: FHIR-53878
- **Related Page(s)**: REST (http)
- **Related Section(s)**: 3.2.0.7.1
- **Related URL**: https://hl7.org/fhir/6.0.0-ballot4/http.html#cdelete
- **Reporter**: DanielSchuster
- **Resolution**: Persuasive with Modification
- **Resolution Description**: Will update the http page:
* Split {{conditional-delete-single}} and {{conditional-delete-multiple}} into two sections under {{conditional-delete}}
* In {{conditional-delete}}, describe the difference between {{single}} and {{multiple}} variants and their syntaxes
* Add {{DELETE [base]/[type]/[id]}} with {{If-Match}} header as a valid pattern for {{conditional-delete-single}}
* Clarify that {{If-Match}} is only valid for {{conditional-delete-single}}
* Clarify that {{DELETE [base]/[type]?_id=[id]}} is a valid variant of {{single}} that servers MAY support
* Clarify in the {{conditional-delete-multiple}} that {{If-Match}} is not supported - if the caller has an ETag, they have the ID of the resource.
- **Status**: Resolved - change required
