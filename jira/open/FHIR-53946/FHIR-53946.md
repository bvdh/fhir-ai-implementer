# FHIR-53946: Proposed New Mechanisms

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/12/26
- **Description**: A set of profiling mechanisms is proposed to improve FHIR conformance clarity, enforce security-related constraints, and support implementer introspection.

1.1 Recursive Constraints on Element.id and Element.extension

Introduce a recursive constraint mechanism applicable to both Element.id and Element.extension.
These rules should only apply to the concrete elements; and referenced entities are explicitly excluded.

If the maximum recursion depth is defined as R, an element A with:

	.id constrained at R0, and
	.extension constrained at R1



indicates that:

	Element A SHALL  have an id without any its own children under the .id.
	Element A MAY carry extensions.
	Extensions under element A SHALL contain any child elements including extensions directly for only one more layer.
	Any .id is a special element and should be treated as an integral component of its parent element.



This mechanism SHOULD be accompanied by coded introspection metadata (e.g., information|warning|error|fatal) to guide implementers in detecting and handling constraint violations or exceptional conditions.   

1.2  Considerations for Conformance validation for a FHIR IG

A use-case–specific FHIR Implementation Guide (IG) may be applied to serve conformance purposes.

	The IG SHALL conform to its parent FHIR resource profiles.
	Any resource instances produced under the IG SHALL conform to the IG.



Such an IG MAY support both sending and receiving endpoints, with primary emphasis on the sending endpoint.

Current FHIR conformance mechanisms do not sufficiently support comprehensive conformance validation. Additional mechanisms are therefore proposed to address this limitation.          

1.2.1 Explicit Conformance Section in StructureDefinition and  elementDefintion.

Introduce an explicit conformance section within StructureDefinition to enumerate required elements in a clear and explicit, while preserving FHIR’s flexibility and extensibility for other scenarios (e.g., break-glass use cases).

Proposed mechanisms include:

	A conformance section that lists all required elements, whether defined in the differential or inherited from parent profile.
	Explicit handling unspecified elements, using complementary strategies:
  - AllButTheRest: A default policy applied to all elements, with support for element-level overrides.
  - TheRestOnly: A focused policy that constrains only those elements not explicitly listed collectively.
	Introspection messaging with categorized severity levels(information|warning|error|fatal) is applied either individually or collectively.         



1.3 Handling of  Elements with specific tags (modifier, summary, mustSupport, obligation)

If elements marked with modifier, summary, mustSupport, or obligation are absent from an instance, an explicit, computable message SHOULD be generated, either individually or collectively, to avoid implicit semantic assumptions.
This functionality is particularly helpful during conformance validation phase, and may be disabled in  fully deployed systems. Additioanlly, it could be considered as component of the metadata resource.

Example AllButTheRest policy:

AllButTheRest.element.tag.value = modifier|summary|ms|obligation, 
AllButTheRest.element.tag.scope  = any|rest,
AllButTheRest.element.tag.handle =inclusion|exclusion,
AllButTheRest.element.tag.description ="
exclusion:
When  tagged elements are absent, its corresponding semantic functions should not be applied either implicitly or explicitly.

Inclusion: 
When  tagged elements are absent , their corresponding semantic functions should still be considered,  and a default, explicit resolution mechanism is provided.
",
AllButTheRest.element.tag.inclusionSolution= "A coded value with an explicitly bound valueset."
AllButTheRest.element.tag.messageLvl=information|warning|error|fatal.
