# FHIR-53835: Rewrite Introduction to Signature (aka Digital Signatures) Page

## Metadata

- **Assignee**: Unassigned

- **Description**: The Signature page is titled "Digital Signature" and focuses almost entirely on digital signatures.  However, the page is referenced as "Signatures" and briefly documents other kinds of electronic signatures.
	The term "electronic signatures" is used incorrectly.
	Documentation on  non-digital electronic signatures is absent.
	Focus on JWS and XML, ignoring the widely used CMS



To correct these issues, I propose rewriting the introductory section to: 

	Title the page "Signatures"
	Create an introduction that covers all types of Electronic signatures
	Use the term "electronic signature" correctly.
	Include a section on non-digital electronic signatures in FHIR
	Include a section on digital signatures
	Add CMS into the conversation
	Summarize the sections that follow.

Signatures

FHIR Support signature functionality through the Signature datatype, the signature element is used in Bundle, Contract, and Provenance, and in the QuestionnaireResponse Signature extension.
Electronic Signatures:

Electronic signatures range from simply typing a name or inserting an image of a handwritten signature to applying a digital signature that uses cryptographic proof to show the content has not changed. All digital signatures are electronic signatures, but not all electronic signatures are digital signatures. This specification provides the data structures for non-digital signatures but leaves implementation details and verification processes to local policy. The use of FHIR digital signatures is discussed in the following sections.

Examples of FHIR non-digital signatures (omitting other Signature datatype fields for brevity):

	Hand-typed name




	Signature.data: The base64-encoded hand-typed name
	Signature.sigFormat: the mime type for the image file (e.g, text/plain)




	Scanned image of handwritten signature




	Signature.data: The base64-encoded JPG of the image.
	Signature.sigFormat: the mime type for the image file (e.g, image/jpg)




	Checking a box: The “proof” is really the audit trail showing a specific authenticated user clicked a specific checkbox at a specific time. Therefore, the Signature.type, Signature.who, and Signature.when are the critical elements, and Signature.data and Signature.sigFormat are not used.



Digital Signature

FHIR Resources are often parts of a Medical Record or are communicated as part of formal Medical Documentation. Digital signatures are sometimes needed to cryptographically bind a signature so that the receiving or consuming actor can verify authenticity, integrity, and rely on non-repudiation of the FHIR content.

There are several digital signature methods, including CMS, JWS, and XMLDSig, each with many options. To provide implementations with the capabilities to support the set of local policies, implementation guides are required to specify how signatures should be created when authoring FHIR content and what actions should be taken to establish full verification when checking signatures. The goal of this section is to provide implementations with the capabilities needed to make the actual signature verification process interoperable.

This specification defines FHIR canonicalization methods and how to identify them. It documents rules and guidelines for using the Signature data type for JWS, XML, and CMS signature methods and their profiles. It provides guidance for signing FHIR Bundles using the Provenance resource.
