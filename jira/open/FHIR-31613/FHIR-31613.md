# FHIR-31613: Clarification on representing include/revInclude 3rd param in CapabilityStatement

## Metadata

- **Assignee**: Corey Spears
- **Created**: 3/24/21
- **Description**: For the searchInclude documentation in a CapabilityStatement, they are expected to be the actual parameter values supported (i.e. "Coverage:payor") but for declaring support for the optional 3rd parameter (i.e. "Coverage:payor:Patient") would that whole value also be expected to be in the CapabilityStatement? or would the root Coverage:payor be enough to imply that optional parameter is also supported.

Clarification is needed on how to represent these include/revInclude values https://chat.fhir.org/#narrow/stream/179166-implementers/topic/CapabilityStatement.20searchInclude.20documentation
- **Grouping**: [CapabilityStatement2]
- **Issue Type**: Change Request
- **Key**: FHIR-31613
- **Related Artifact(s)**: CapabilityStatement (Conformance)
- **Related URL**: https://www.hl7.org/fhir/capabilitystatement-definitions.html#CapabilityStatement.rest.resource.searchInclude
- **Reporter**: Ryan Moehrke
- **Resolution**: Persuasive with Modification
- **Resolution Description**: In CapabilityStatement2, we will:
 * ensure there's an ability to say that you support all (or none) of the Resource filters
 * make clear that declaring support for the non-resource qualified version does not imply support for the resource-qualified version
 * that servers that don't support the filtered version but do support the non-filtered version may choose to return either the non-filtered version or ignore the _include.  Whatever decision is made, what is returned SHALL be reflected in the 'self' link
- **Status**: Resolved - change required
