# FHIR-54070: Subscription.filterBy.comparator seems redundant

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/13/26
- **Description**: Subscription.filterBy.comparator seems redundant; the comparator should be included in Subscription.filterBy.value

 

1) The long description for filterBy.value shows this explicitly:

http://build.fhir.org/subscription-definitions.html#Subscription.filterBy.value



 
The literal value or resource path as is legal in search - for example, Patient/123 or le1950.





2) filterBy.comparator is only 0..1, which wouldn't support all "or logic" situations.

For example, if filtering for only extreme values, you might use both lt and gt.

filterBy.value = lt50,gt100

 

It seems like filterBy.comparator should be removed, and for filterBy.value to match the entire query parameter value (for search param-based filters), including the comparator
