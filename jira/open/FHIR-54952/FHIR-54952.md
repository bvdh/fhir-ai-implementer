# FHIR-54952: Definition Formatting unclear

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/20/26
- **Description**: The style of representing the language's Definitional blocks , no convensions are defined in the specification to explain how to read these blocks.

(see how fhirpath does it here https://build.fhir.org/ig/HL7/FHIRPath/#conventions)

Also Examples are not easily differentiated from the definitional blocks.

Some of the mixed formats that need definition (how can we understand what is literal characters vs features such as optional/repeating etc):

	/// name{.property}* = {value}
	let &#91;name&#93; = &#91;fhirpath expression&#93;;
	group &#91;group-name&#93; (inputs) (extends &#91;other-group&#93;) (&lt;&lt;stereotype&gt;&gt;) {
	&#91;mode&#93; &#91;name&#93;( : &#91;type&#93;)



And blocks like this one in transform Rules, looks like it's defining the entire block, but I don't think this is a definition, or an example, especially when it is followed immediately by an example.

	src_context.field as new_variable where (condition) -&gt; tgt_context.field = create(&#91;type&#93;) as new_variable then &#91;details&#93; "name";



This appears to be implying that the type is optional, unless the square brackets are meant to be documenting identifiers? If they are, then why aren't src_context and field like it?

Would using fragments of an ANTLR style grammar be a better choice for the definitional parts? That would take out the ambiguities.
