# FHIR-51728: Compliant compartment deviation is unclear

## Metadata

- **Assignee**: Unassigned
- **Created**: 8/12/25
- **Description**: At present, compartment definitions can only be defined by HL7 International. This is because their existence creates significant impact on the behavior of servers.
	Although compartment definitions can be a useful foundation for access control, they may not be sufficient for this purpose. Resources outside the compartment might still be appropriate to be made available or subject to compartment restrictions even though they are not formally part of the compartment (see below about access rules).



...

The full definitions of these compartments are published as CompartmentDefinition resources. Servers typically do not support the full definition of a compartment, and are not required to. Systems may publish CompartmentDefinition resources so that other systems may make use of compartments properly.
The description of CompartmentDefinition indicates that while only HL7 may define CompartmentDefinitions, implementations are not bound to that definition (although there is no computable way of expressing what they do implement, since only HL7 may define compartmentDefinitions.)

In discussion, there was an assertion that the ability to implement less than a full compartment definition is to allow a server to still support a compartment even if a particular resource type is not supported by the system. (E.g., A system that implements the Patient compartment, doesn't need to include Account in the compartment if they don't generally support account.) However, this intent is not clear in the text, and it isn't clear if there may be other reasons for supporting less than the full definition. It also isn't clear if supporting more than the definition is permitted, although there is text in the resource that implies that.

There was also an assertion that it may not be feasible for some systems (e.g. facades) to implement the full definition, even if all resources are supported outside the compartment.

Clarify under what circumstances a compliant implementation is permitted :

	To not implement a compartment when the base resource is supported.
	To not include in a compartment, a resource type listed in that compartment definition, when the resource type is supported.
	To not use a search parameter listed in the definition to determine membership of a resource in a compartment, when the search parameter is supported on the resource directly.
	To use a search parameter not included in the compartment definition, or another method, to determine membership of a resource in a compartment.
	To include a resource type not included in the compartment definition, in a compartment.
	To not include a resource instance in the compartment when it otherwise would be expected in the compartment (other than "security").
