# FHIR-56879: Define how translatable behaves for extensions

- **Description**: At present, the expectation is that translatable is defined for a resource or data type element and is fixed from that point forward.  However, this is a problem for extensions, where some extensions of type string might be translatable, while others won't be.  Grahame's thought is that we might handle this the same way we handle isModifier, which is allowed to be set on some extensions, but not others.  (Though that can only happen at the root, while translatable would be for values, and possibly for values within sub-extensions.)

While we're at it, we might contemplate whether extensions are allowed to define 'meaningWhenMissing' and/or 'orderMeaning', because they could theoretically vary for extensions as well.
