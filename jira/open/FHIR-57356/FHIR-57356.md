# FHIR-57356: Restore Bundle.link.relation to allow extension URIs (align with RFC 8288)

## Metadata

- **Assignee**: Unassigned
- **Created**: 5/19/26
- **Description**: Summary

Roll back the R5 change that narrowed Bundle.link.relation from string (R4) to code with a required binding to the IANA Link Relations registry. Restore the ability to use extension relation types (full URIs) so FHIR matches HTTP link relation semantics.
Background

After R4, we proposed hard-limiting relation to IANA-registered types. HTTP itself does not work that way.

RFC 8288, Section 2.1 ("Relation Types") defines two parallel forms:

	§2.1.1 — Registered Relation Types: simple token syntax (reg-rel-type, lowercase ALPHA plus a few other chars). MUST be registered in the IANA Link Relations registry; registration policy is Specification Required.
	§2.1.2 — Extension Relation Types: URIs (ext-rel-type), used to avoid collisions when you need a relation outside the registry. The URI does not have to be dereferenceable — it is just an identifier.



R4 mirrored this by typing Bundle.link.relation as string, which accepted either form. R5 dropped the extension form entirely. We no longer allow non-registered relationship types — we diverge from HTTP.
What the record shows

Our JIRA discussion at FHIR-27787 included this comment from @Lloyd McKenzie:
We can't change the data type as that would be a breaking change.
But then we did in fact change the data type (string → code). Per the git history (commit d1672bb7c2, 2022-09-04), the type change was driven by a publisher-tooling limitation — the build did not allow bindings on string types — not by a committee decision to expand the scope of FHIR-27787.

The community consultation on Zulip had no meaningful participation: the outreach message, one clarifying question, and one reply. The scope presented to the community was "adding the binding as a clarification" — not changing the datatype.
Proposal

Revert Bundle.link.relation to allow extension relation types, matching RFC 8288 and R4 behavior. Options, in order of preference:

	Change type back to string with a documented expectation that the value is either an IANA-registered token or a URI per RFC 8288 §2.1.2. Drop the required binding; keep an example or preferred binding pointing at the IANA set.
	If keeping code is required by tooling, broaden the binding to extensible or preferred so extension URIs remain conformant.



Either path restores interoperability with the wider HTTP/Web Linking ecosystem and undoes a breaking change that was never actually agreed to.
