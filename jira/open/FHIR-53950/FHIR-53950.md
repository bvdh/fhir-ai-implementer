- **Description**: Incorrectly states that we can only map/use FHIR.Quantity if using UCUM units.
You should also be able to use the FHIRPath calendar units codesystem: http://hl7.org/fhirpath/CodeSystem/calendar-units
Update to include both code systems (and link to the fhirpath spec).
 
Why are we repeating the quantity handling notes here in the fhir page? Remove the duplication of the fhirpath table. https://build.fhir.org/ig/HL7/FHIRPath/#time-valued-quantities if this is so important, at least refer to the relevant part of the original source specification and note that it is replicated here for emphasis. (also remove the dragon, as that content is in the fhirpath spec - interestingly has no dragon there...)

We should NOT be converting codesystems unless explicitly requested to do so.

Please indicate what should occur if using a quantity that is not one of the 2 supported unit systems. I expect that this should just return empty for these cases, but this is not super clear to me if this is right, or if it should report an exception to the caller and terminate the expression.

Consider if these 3 sections (and the above time valued quantities table) from the fhirpath spec are worth including as references/links 
https://build.fhir.org/ig/HL7/FHIRPath/#datetime-arithmetic
https://build.fhir.org/ig/HL7/FHIRPath/#toquantityunit--string--quantity
https://build.fhir.org/ig/HL7/FHIRPath/#quantity-equality
