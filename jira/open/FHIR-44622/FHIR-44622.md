- **Description**: Summary: This change request seeks to allow for more granularity for expressing the interaction codes associated with profiles in a capability statement. Currently, you can only express the supported profiles and the interaction codes as a group. A more precise way is to add the capability to express individual interaction codes for a given profile.

For each exchanged resource referenced in the capability statement, the current method allows to assign a list of interactions (Read, Write and other codes from Interaction Codes value set). However, once selected, the interaction codes are applied to all the supported profiles for a given resource type.

Table 1 illustrates how the interaction codes are assigned to a particular resource type. In this case, the Read and Search operations are applied to all the supported profiles.

### 2. Lloyd McKenzie - 3/25/24
Security: Public

Based on our understanding, this has nothing to do with ElementDefinition obligations.  This is about having distinct profiles for different RESTful interactions (much as we can already have distinct profiles for the inputs and outputs of custom operations).  We are in favor of there being some extension mechanism for allowing a 'profile' to be declared on each interaction for a resource (e.g. a different profile for 'search' than for 'history').  However, we'd ideally like this to be handled as part of CapabilityStatement2.  We're tagging it for consideration as part of that project.  If that approach turns out to not be viable, we can revisit and perhaps define a standard extension.
