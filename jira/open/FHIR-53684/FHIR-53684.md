- **Description**: Some implementation guides are not meant to be implemented directly, but rather serve as a "base" for derivative implementation guides to inherit from.  Some examples include:

	Clinical Order Workflow (COW)
	Many national "base" IGs.



There are also some other "features" that are not meant to be implemented directly, but aren't "ImplementationGuides".  One primary example is SMARTv2 scopes.

 

It would be useful to establish some method of identifying these "abstract" artifacts, so that it is clear what content is intended to be implemented by software developers, vs. what content is meant to be building blocks for derivative IG authors.

 

In my head, I'm thinking of this like the "abstract" keyword in C#.

 

I'm writing this against ImplementationGuide, but I acknowledge that there are some "Implementation Guides" that don't use the ImplementationGuide resource (e.g., SMARTv2 scopes), so it won't catch everything.  I'm open to other options for how to flag such content.
