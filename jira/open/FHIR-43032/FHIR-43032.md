- **Description**: When defining a profile, we want to link it to the Logical Map that represents it.

The detailed mapping - at element level - between the LM and profile can be established with ConceptMap,

This request is for a simple indication that "this profile implements logical model X" This could be expressed by either/any of:

	a new extension on the profile
	a ConceptMap (mapping the top elements)
	mapping inside the resource metadata



 

If the preferred approach is a ConceptMap, this may be interesting to document in the logical models page.

If the preferred approach is an extension, one should be defined.

### 8. Alexander Henket - 5/20/24
Security: Public

We have looked at various ways to do this years ago and settled for mappings inside the profile. The downside is that updates to that mapping, constitute a change in the profile. This is fine if the profile is ours, but a bit of a waste when the profile is not, e.g. IPS.  We will then create a profile on top of that external profile for the purposes of adding those mappings.

Having something outside of the profile that merely points to the profile, was considered but how to then offer implementers a view on how these separate artifacts relate was unsolvable back in the day. Having the mapping and the profile version independently would be a good thing.

Having a derived profile that only lists the mappings, is somewhat the same as what the proposed ConceptMap would do, would it not? The added benefit is that you'll have more context and validation than possible with a ConceptMap.

### 10. Lloyd McKenzie - 5/20/24
Security: Public

Proposal

	We should add a decision tree in StructureDefinition R6 that outlines the choices when wanting to map content to a profile or logical model




	If you want to be able to computably convert between two structures and check validity of those maps - StructureMap
	If you're saying "this corresponds to concept x (where that's a code from a code system or data element from a library), use ElementDefinition.code
	Use ElementDefinition.map, ideally with a language of FHIRPath if you want rougher 'equivalence' mappings, using map.comment if you need to add guidance beyond what you can say in FHIRPath.  (Alternative would be ConceptMap?)



2. We want the tooling to:

	create a list of models that map to this element and a list of models this maps to
	Validate that maps are valid paths.



Jose Costa-Teixeira to produce an example we can use to see what this looks like and try creating tooling.  Will do a bake-off between the ElementDefinition.map and the ConceptMap approaches.
