# FHIR-53998: Clarify expected behavior for multiple _include:iterate parameters

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/12/26
- **Description**: What is the expected behavior from servers if a request includes multiple _include:iterate parameters?

 

For example, if a request has the following two parameters in addition to an unmodified _include parameter
___include:iterate=ResourceA:elementE&amp;{{{}include:iterate=ResourceB:elementF}}
Should servers:

	(1) Take the initial pool of included resources, apply the first of these {{iterate }}parameters to get a larger pool of included resources, and then apply the second parameter on the larger pool?
	(2) apply all iterate parameters to the initial pool, and then apply all parameters again to the larger pool, and so on for however many iterations they support?



In other words, should the order of {{iterate }}parameters in the request matter?

 

I expect that (2) is what they're expecting servers to do since the order of search parameters is more generally not supposed to affect the outcome of the request.

But I'm not sure if there would ever be a use case for (1) such that it would be helpful to clarify what the expected behavior should be.
