# FHIR-53182: Special rules for .meta.security coding?

## Metadata

- **Assignee**: Unassigned
- **Created**: 10/20/25
- **Description**: Why is the "Note" on Security Labels? It seems to stem from long ago (FHIR-20632) when .security was changed from uri to coding; but the note seems problematic, especially as we get more strict on code system version processing. I don't think the note is proper, and I am unsure that the validator allows deviations from code display value. Would seem best to remove the note.
2.1.27.5.3.9.3 Security Labels 

A security label is attached to a resource to provide specific security metadata about the information in the resource. For further information, see Security Labels.

Note, it is valid to create two security labels with the same (system+code) but different .display or .version. Furthermore, .display or .version are not taken into account when evaluating security labels.
