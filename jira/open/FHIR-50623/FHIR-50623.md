# FHIR-50623: Can Clients Request Different Content level via Subscription.$event?

## Metadata

- **Assignee**: Unassigned
- **Created**: 5/2/25
- **Description**: This is a technical correction + a question:

 

1) The content parameter has this documentation referring to the backport content value set - that should probably refer to the R6 valueset instead?

Requested content style of returned data. Codes from backport-content-value-set (e.g., empty, id-only, full-resource). This is a hint to the server what a client would prefer, and MAY be ignored.

 

2) Can/should servers support clients requesting different content level on this operation (compared to what the client requested on creating the subscription)? i.e. is this a way to send notifications with empty/id-only, and only serve up full-resource if queried? If so - would the full-resource be the "snapshot" in time from when the notification was sent, or the current state (as if querying the FHIR server in a read with the ID)?
