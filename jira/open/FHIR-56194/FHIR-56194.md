# FHIR-56194: Clarify language for GET invocation of operations

## Metadata

- **Assignee**: Unassigned
- **Created**: 3/30/26
- **Description**: The operations framework allows operations to be invoked with GET if:

&gt; the only parameters being passed are ones whose data type has a corresponding search type (i.e., datatypes like Identifier or Reference which have a search type of 'token' are permitted, while Address and HumanName are not)

However, there are at least two places where the "primitive type" characterization is used:

After the first example in the "Executing an Operation" section (https://build.fhir.org/operations.html#executing)

&gt; When an operation has affectsState = false, and the parameters are all primitive datatypes with no extensions (as is the case with the example above), it MAY be invoked using GET as well....

Suggest that be updated to:

&gt; ... and the parameters are all types that have a corresponding search type...

Also here (https://build.fhir.org/operations.html#request):

&gt; If all the parameters for the operation are primitive types and the operation has affectsState = false, the operation is also able to be invoked by performing an HTTP GET operation...

Suggest that this be clarified as well to:

&gt; If all the parameters for the operation are primitive types (or have a corresponding search type...

Also suggest that we be explicit about what "has a corresponding search type" means, is there a table somewhere that lists the corresponding search type for each FHIR data type?
