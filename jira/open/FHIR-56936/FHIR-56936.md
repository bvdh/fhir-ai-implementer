# FHIR-56936: Duplicate sliceName "otherTherapy" in snapshot of profile-ClinicalUseDefinition (hl7.fhir.uv.xver-r5.r4#0.1.0) violates ElementDefinition.id uniqueness

Affected artifact

	Package: hl7.fhir.uv.xver-r5.r4
	Version: 0.1.0
	Profile: profile-ClinicalUseDefinition
	Canonical URL: http://hl7.org/fhir/5.0/StructureDefinition/profile-ClinicalUseDefinition



Problem

The snapshot of profile-ClinicalUseDefinition contains two ElementDefinition entries with identical id and identical sliceName, both sitting on the same slicing level (path = Basic.extension):
 
 
json
 {{
{"id": "Basic.extension:otherTherapy","path": "Basic.extension","sliceName": "otherTherapy",  ...}
}}
This element appears twice in the snapshot. The two entries are not duplicates in a semantic sense — they correspond to two distinct R5 source paths:

	ClinicalUseDefinition.contraindication.otherTherapy
	ClinicalUseDefinition.indication.otherTherapy



When the cross-version extensions are generated, both R5 paths are flattened onto Basic.extension as cross-version extensions, and the sliceName is derived from the last R5 path segment only (otherTherapy). Because both source paths share that last segment, the resulting slices collide on id and sliceName.

The same pattern likely affects other xver profiles wherever an R5 element name reoccurs under multiple parent elements within the same resource.
Why this is a problem

This violates the FHIR rules for slicing and ElementDefinition.id construction:

	Within a single slicing context, sliceName must be unique. The ElementDefinition.id is constructed as path + ":" + sliceName, so duplicate sliceNames produce duplicate ids.
	ElementDefinition.id must be unique within a StructureDefinition (invariants sdf-9 / eld-2).
	As a consequence, tooling that processes the snapshot by id (e.g. for navigation, mapping, validation, or rendering) cannot distinguish the two slices. The only differentiator left in the current output is the definition text, which is not a reliable machine-readable key.



Suggested fix

Disambiguate the sliceName so that each slice on Basic.extension is uniquely identifiable. Two reasonable approaches are available within the FHIR specification:

Option 1 — Distinct slice names based on the originating R5 parent (recommended, simpler):
 
 
 {{sliceName: "contraindicationOtherTherapy"   →   id: Basic.extension:contraindicationOtherTherapy
sliceName: "indicationOtherTherapy"         →   id: Basic.extension:indicationOtherTherapy}}
Option 2 — Re-slicing syntax with /:
 
 
 {{sliceName: "otherTherapy/contraindication"   →   id: Basic.extension:otherTherapy/contraindication
sliceName: "otherTherapy/indication"         →   id: Basic.extension:otherTherapy/indication}}
Option 2 is more formally aligned with the FHIR profiling specification's re-slicing pattern but requires a base otherTherapy slice to be defined first. Option 1 is the more pragmatic fix given the flat extension structure used in the xver packages.
Scope and recommendation

Because this is most likely a generator-level pattern rather than a one-off issue with this specific profile, an audit of all xver profiles (across all version pairs, not only R5↔R4) for duplicate (path, sliceName) pairs in their snapshots would be worthwhile. The fix probably belongs in the cross-version extension generator, where the sliceName derivation logic should disambiguate when the last R5 path segment alone is not unique within the target's flat extension list.
