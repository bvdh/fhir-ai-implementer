# FHIR-55742: documentation of 1e2 search behavior

## Metadata

- **__EMPTY**: MichaelKittrell
- **__EMPTY_1**: Public
- **__EMPTY_2**: 3.1.1.4.6 number 

Searching on a simple numerical value in a resource. Examples:



&#91;parameter&#93;=100
Values that equal 100, to 3 significant figures precision, so this is actually searching for values in the range [99.5 ... 100.5)


&#91;parameter&#93;=100.00
Values that equal 100, to 5 significant figures precision, so this is actually searching for values in the range [99.995 ... 100.005)


&#91;parameter&#93;=1e2
Values that equal 100, to 1 significant figures precision, so this is actually searching for values in the range [95 ... 105)


&#91;parameter&#93;=lt100
Values that are less than exactly 100


&#91;parameter&#93;=le100
Values that are less or equal to exactly 100


&#91;parameter&#93;=gt100
Values that are greater than exactly 100


&#91;parameter&#93;=ge100
Values that are greater or equal to exactly 100


&#91;parameter&#93;=ne100
Values that are not equal to 100 (actually, in the range 99.5 to 100.5)

- **Description**: Documentation for "number" is very wrong in the case of 1e2 and isn't as explicit as it should be in 100 and100.00 

1e2



&#91;parameter&#93;=1e2
Values that equal 100, to 1 significant figures precision, so this is actually searching for values in the range [95 ... 105)





This is incorrect, 50 to but not including 150 are both represented as 1e2 to 1 significant figure.  This seems corrected in v5 (but is also incorrect in R4B

https://hl7.org/fhir/R4B/search.html (wrong same as above)
http://hl7.org/fhir/R5/search.html (mostly correct).  



&#91;parameter&#93;=1e2
Values that equal 100, to 1 significant figure precision, so this is actually searching for values in the range [50 ... 150)






Also, I'm concerned about the other entries for 100 and 100.00

5 rounds in normal math so listing the range of 99.5 ... 100.5 suggests that 100.5 will be in the criteria...  100.5 should be rounded up to 101 and NOT be in the "100" parameter range of valid values... unless math has changed or FHIR is using by default the HALF_EVEN rounding mode (I don't know enough to know what FHIR's default is).  If it is using half_even, i think that should be explicity stated.
