# FHIR-55991: Clarify package-based canonical resolution

## Metadata

- **Assignee**: Unassigned
- **Created**: 3/2/26
- **Description**: The section on canonical version matching currently includes this sentence:

&gt; If there is no stated version for the reference, or the version is a wildcard (see below), and if the reference is found in a resource in an NPM Package, then any application looking for the correct version SHALL first look through the package and its dependencies when resolving the reference.

However, the following paragraph indicates:

&gt; Some contexts of use require that applications consult a manifest that provides a default or mandatory version to use for each canonical resource. In these contexts, the application SHALL use the version specified in the manifest when resolving the reference.

But since a canonical resource in a manifest-based context may also be defined in an NPM package, these two requirements are conflicting. Suggest correcting this by changing the first sentence to:

&gt; ... when resolving the reference, unless another explicit version resolution method is provided (e.g. manifest-based, discussed below).
