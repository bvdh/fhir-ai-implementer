# FHIR-57427: authoredOn is named confusingly

## Metadata

- **Assignee**: Unassigned
- **Created**: 5/22/26
- **Description**: "authoredOn" seems to be named confusingly given that is defined in ServiceRequest as the date the request was "signed" (not written or authored). This gets even more confusing now that the O&amp;O working group has decided to create a separate technical 'creationDate'. I would suggest renaming "authoredOn" to 'authorizedOn' to clarify the difference between these two elements as well as the intention of this element on its own.
- **Issue Type**: Change Request
- **Key**: FHIR-57427
- **Related Artifact(s)**: ServiceRequest

Request Pattern
- **Related Section(s)**: Pattern content
- **Related URL**: http://build.fhir.org/request.html
- **Reporter**: OlavTrauschke
- **Resolution**: Not Persuasive with Modification
- **Resolution Description**: There are two challenges:
 # Making this change is a breaking change and we're very reluctant to do that merely to provide more clarification
 # Not all 'request' resources represent authorizations.  If the intent is 'proposal' or one of its specializations, 'directive', or 'option', the transition to active isn't actually authorizing action.

What we can reasonable do is this:

Change the short description from "When request was created/transitioned to active"

to "When request was created/transitioned to active (e.g. signed/authorized)"

That helps to include the key term in something the reader of the spec will always see, without breaking the spec and without creating a situation where we'd need different elements for requests that are authorizations and those that aren't.

We will change the short description as described above.  We will also add a new "alternate name" of "authorizedOn".
