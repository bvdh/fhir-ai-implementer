# FHIR-57383: Resource.id / Patient.id `structuredefinition-fhir-type` Regression

## Metadata

- **Assignee**: Unassigned
- **Created**: 5/20/26
- **Description**: Summary

Between the published R6 ballot3 and ballot4 artifacts, the generated Patient.id element changed from:



"valueUrl": "id"



to:



"valueUrl": "string"



This appears to be a Kindling generation regression. Patient.id is inherited from Resource.id, and Resource.id still has structuredefinition-fhir-type = id in R6 ballot4.

The likely cause is Kindling commit:

https://github.com/HL7/kindling/commit/62a8c188e5ff9dcee8bbfe6ac10a7670d934abed

Why this looks like a regression

Patient.id is inherited from Resource.id. In both ballot3 and ballot4, the generated Patient.id element includes:



"base": {
  "path": "Resource.id",
  "min": 0,
  "max": "1"
}



The base Resource.id remains:



"valueUrl": "id"



but the inherited Patient.id became:



"valueUrl": "string"



So the generated derived snapshot no longer preserves the fhir-type of the inherited resource id.

This is also inconsistent with the recent release pattern:

Kindling change that explains the regression

The relevant Kindling file is:



src/main/java/org/hl7/fhir/definitions/generators/specification/ProfileGenerator.java



Before commit 62a8c188, top-level .id paths were assigned fhir-type id based on path depth:



} else if (Utilities.charCount(path, '.') &lt; 2) {
  ExtensionUtilities.addUrlExtension(tr, ExtensionDefinitions.EXT_FHIR_TYPE, "id");
} else {
  ExtensionUtilities.addUrlExtension(tr, ExtensionDefinitions.EXT_FHIR_TYPE, "string");
}



After commit 62a8c188, the rule requires isResource:



} else if (isResource &amp;&amp; Utilities.charCount(path, '.') &lt; 2) {
  ExtensionUtilities.addUrlExtension(tr, ExtensionDefinitions.EXT_FHIR_TYPE, "id");
} else {
  ExtensionUtilities.addUrlExtension(tr, ExtensionDefinitions.EXT_FHIR_TYPE, "string");
}



That change was probably intended to make Element.id-style ids use string, which is reasonable. The problem is that inherited resource-level ids are generated through a path that passes isResource = false.

How Patient.id falls through

Kindling expands inherited ancestor elements using defineAncestorElements(...).

In the same file, that method calls defineElement(...) like this:



ElementDefinition ed = defineElement(
  null,
  p,
  elements,
  child,
  path + "." + child.getName(),
  containedSlices,
  new ArrayList&lt;ProfileGenerator.SliceHandle&gt;(),
  snapshot,
  false,
  dt,
  null,
  defaults,
  false
);



The final argument is isResource, and it is hard-coded to false.

So when inherited Resource.id is emitted as Patient.id, the values are effectively:



path = Patient.id
isResource = false
dot count = 1



Before commit 62a8c188, dot count alone was enough to assign fhir-type id.

After commit 62a8c188, isResource must also be true, so Patient.id falls through to fhir-type string.

Nested ids such as Patient.contact.id should still be string, because their path has more than one dot. The regression is specifically that inherited top-level resource ids lost the resource context.

Likely fix direction

The fix is likely in Kindling.

The generator should distinguish:


	top-level resource ids, such as Patient.id, Observation.id, DomainResource.id: fhir-type {
Unknown macro: {id} 
}
	internal element ids, such as Patient.contact.id, Element.id, BackboneElement.id: fhir-type {
Unknown macro: {string} 
}



One possible fix is to preserve resource context when calling defineElement(...) from defineAncestorElements(...), instead of hard-coding isResource = false for inherited ancestor elements.

The existing dot-count condition can still protect nested ids:



isResource &amp;&amp; Utilities.charCount(path, '.') &lt; 2



but inherited resource-level ids need to reach that branch with isResource = true.

Related tracker context

Relevant Jira items:


	FHIR-25262: resource ids should use fhir-type id, not string.
	FHIR-25274: documents the System.String plus structuredefinition-fhir-type mechanism.
	FHIR-41183: the issue referenced by the Kindling commit that changed this rule.
	FHIR-45989: discusses the distinction between Resource.id and Element.id.



The historical distinction appears to be Resource.id versus Element.id, not Resource.id versus inherited resource ids like Patient.id.
