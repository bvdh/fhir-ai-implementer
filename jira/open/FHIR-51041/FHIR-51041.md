# FHIR-51041: Additional resources is a misleading name

## Metadata

- **Assignee**: Unassigned
- **Created**: 6/8/25
- **Description**: The name "additional resources" may suggest that anyone can make their own resources (which we would actually call custom resources). That is certainly the impression I had initially. 

I suggest either officially calling them "additional core resources" (which could still be abbreviated to "additional resources" in conversation, or later slides in a deck), or "interim resources" (which says that the difference is mostly about timing rather than about stewardship or being additional "to" FHIR).
- **Issue Type**: Change Request
- **Key**: FHIR-51041
- **Related Page(s)**: (methodology)
- **Reporter**: Rik Smithies
- **Resolution**: Not Persuasive with Modification
- **Resolution Description**: We feel that introducing 'core' as a qualifier would have the reverse effect - it would suggest that there is some 'other' permitted non-core set of additional resources that could be defined, which is not the case.  The section is quite clear about the fact that this process is governed by HL7 and the intent is always that they'll eventually be added to core.  (Note, however, that in theory HL7 could permit another organization such as IHE to curate an additional resource - the methodology doesn't prohibit this - so long as HL7 approves and registers the name.)

Two changes:

1. We will fix the lower-case 'd' in "Additional Resources defined Outside This Specification"

We will add a separate line at the end of the "Additional Resources defined Outside This Specification" section that says:

"The introduction of 'custom' resource not defined through the process documented here is not conformant with the FHIR specification.  Instances communicating resources not defined in this core specification or via the additional resources registration process cannot be called FHIR conformant."
- **Status**: Resolved - change required
- **Summary**: Additional resources is a misleading name
