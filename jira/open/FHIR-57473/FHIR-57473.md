# FHIR-57473: Document the @search() capability

## Metadata

- **Description**: The HAPI FML engine has a @search() capability that is undocumented and I believe was the intent of the mapping support API... (and also update the grammar too)

&#8212;


	
	
		The `search(...)` source clause
	
	



A rule source may use the special form `search(...)` in place of an input variable name. This allows a rule to draw its source items from a query executed by the host application, rather than from data already present in the transform's input variables.


	
	
		
		
			Syntax
		
		
	
	



```
source ::= ( &lt;variable&gt; [ '.' &lt;element&gt; ] | 'search' '(' &lt;fhirpath&gt; ')' )
           [ ':' &lt;type&gt; ]
           [ &lt;min&gt; '..' &lt;max&gt; ]
           [ 'default' '(' &lt;fhirpath&gt; ')' ]
           [ &lt;listMode&gt; ]              // first | last | not_first | not_last | only_one
           [ 'as' &lt;variable&gt; ]
           [ 'where' '(' &lt;fhirpath&gt; ')' ]
           [ 'check' '(' &lt;fhirpath&gt; ')' ]
           [ 'log' '(' &lt;fhirpath&gt; ')' ]
```

The token `search` is only treated as a search clause when it is immediately followed by `(`; otherwise it is parsed as an ordinary variable name. The `search(...)` form is mutually exclusive with `.element` — a search source supplies the items directly and has no element path to navigate.


	
	
		
		
			Semantics
		
		
	
	



Evaluation of a `search(...)` source proceeds in two stages:

1. *Build the query string.* The text between the parentheses is a FHIRPath expression. It is evaluated against the variables currently in scope and is required to produce a single `string` value. Typical use is string concatenation that interpolates values from source variables, e.g. `'Patient?identifier=' + ident.system + '|' + ident.value`.

2. *Execute the query.* The resulting string is passed to the host application's transform services (`performSearch` on the `ITransformerServices` interface, or its equivalent in non‑Java implementations). The host is responsible for interpreting the string — conventionally as a FHIR REST search expression — and returning the matching resources. Each returned resource becomes a source item for the rule, in the same way as items produced by an ordinary `context.element` source.

The FML engine itself does not parse, validate, or execute the search string; that responsibility lies entirely with the host. As a consequence, rules that use `search(...)` are inert when the engine is run without a host services implementation, and the search syntax accepted is determined by the host (most implementations accept standard FHIR REST search parameters).

The `:Type`, cardinality (`min..max`), `default(...)`, `listMode`, `as`, `where`, `check`, and `log` modifiers apply to a `search(...)` source in exactly the same way as to a normal source.


	
	
		
		
			Examples
		
		
	
	



Fetch a `Patient` by a known business identifier and bind it to a variable:

```fml
src -&gt; tgt then 
{
  search('Patient?identifier=http://acme.org/mrn|12345') as patient
    -&gt; tgt.subject = create('Reference') as r,
       r.reference = ('Patient/' + patient.id) "findPatient";
}
```

Build the query from data already in scope, requiring exactly one match:

```fml
order.subject as ref -&gt; tgt then 
{
  search('Patient?_id=' + ref.reference.substring(8)) : Patient 1..1 first as patient
    -&gt; tgt.patient = patient "resolvePatient";
}
```

Combine multiple variables into the query:

```fml
src.identifier as id -&gt; tgt then 
{
  search('Patient?identifier=' + id.system + '|' + id.value) as patient
    -&gt; tgt.entry as e, e.resource = patient "byIdentifier";
}
```


	
	
		
		
			Notes for implementers
		
		
	
	




	Because the search string is produced by FHIRPath, all standard FHIRPath escaping and string‑construction rules apply. URL‑encoding of values, if required by the host, is the author's responsibility.
	A host that does not support search should signal this clearly (e.g. raise an error) rather than silently returning an empty list, since silent failure will manifest as a rule that produces no output for reasons that are not visible in the map.




	
	
		
		
			Representation in the `StructureMap` resource
		
		
	
	



When a rule source is authored as `search(...)` in FML, it is recorded in the underlying `StructureMap.group.rule.source` element as follows:




 FML                                  
 `source.context` 
 `source.element`                


--------------------------------------
------------------
---------------------------------


 `src.identifier as id`               
 `"src"`          
 `"identifier"`                  


 `search('Patient?_id=' + id.value)`  
 `"@search"`      
 `"'Patient?_id=' + id.value"`   





That is:


	`context` is set to the sentinel value `"@search"`. The leading `@` distinguishes it from any real input variable (variable names cannot begin with `@`), so engines can recognise a search source without ambiguity.
	`element` carries the FHIRPath expression that, when evaluated, produces the search query string. It is stored as the FHIRPath source text, not as a pre‑computed query.
	`defaultValue`, `type`, `min`/`max`, `listMode`, `variable`, `condition`, `check`, and `logMessage` are populated normally and have the same meaning as for any other source.



When serialising a `StructureMap` back to FML, an engine should detect `context = "@search"` and render the source as `search( &lt;element&gt; )` rather than as `@search.&lt;element&gt;`.
