# FHIR-57211: xver: Inconsistent R5 Availability mapping between HealthcareService and PractitionerRole

## Metadata

- **Assignee**: Unassigned
- **Created**: 5/12/26
- **Description**: In `hl7.fhir.uv.xver-r5.r4#0.1.0`, `HealthcareService.availability` and `PractitionerRole.availability` appear to have the same R5/R4 model shape, but the xver mappings treat them differently.

Observed xver behavior

`HealthcareService.availability` is mapped directly to the two R4 sibling backbones:


HealthcareService.availability
  -&gt; HealthcareService.availableTime    equivalent
  -&gt; HealthcareService.notAvailable     equivalent



No generated `extension-HealthcareService.availability` target appears for this element in the package.

By contrast, `PractitionerRole.availability` maps to the same kind of R4 sibling pair, but also keeps an R5 extension:


PractitionerRole.availability
  -&gt; PractitionerRole.availableTime     relatedto
  -&gt; PractitionerRole.notAvailable      relatedto
  -&gt; extension-PractitionerRole.availability wider



The package also includes a generated `StructureDefinition/extension-PractitionerRole.availability` target for this element.

The official xver comment for `PractitionerRole.availability` says the direct mappings "do not cover the following types: Availability", which appears to be the reason the extra extension is generated. But the same reasoning seems applicable to `HealthcareService.availability`.

Why this is surprising

In R5, both source elements have the same datatype and cardinality:


HealthcareService.availability   0..* Availability
PractitionerRole.availability    0..* Availability



Both also have the same mapping identity:


rim:.effectiveTime



In R4, both resources have the same target pattern:


HealthcareService.availableTime   0..* BackboneElement
HealthcareService.notAvailable    0..* BackboneElement

PractitionerRole.availableTime    0..* BackboneElement
PractitionerRole.notAvailable     0..* BackboneElement



The child shapes are parallel as well:


availableTime.daysOfWeek
availableTime.allDay
availableTime.availableStartTime
availableTime.availableEndTime
notAvailable.description
notAvailable.during



There are some known R5/R4 shape differences, such as R5 `Availability.notAvailableTime.description` being optional while R4 `notAvailable.description` is required, and the R5 wrapper being named `Availability` while R4 uses two sibling backbones. But those differences apply to both `HealthcareService` and `PractitionerRole`.

Question

Is there a domain-specific reason why `PractitionerRole.availability` preserves the R5 `Availability` wrapper as an extension, while `HealthcareService.availability` does not?

If the R4 `availableTime` / `notAvailable` pair is considered a complete representation of R5 `Availability`, then it seems like both mappings could be direct/equivalent.

If preserving the R5 `Availability` wrapper is important because splitting into two sibling arrays loses the R5 grouping semantics, then it seems like both mappings should include a generated R5 availability extension.

At the moment, the two cases look structurally equivalent but are treated differently, which makes it hard for downstream generators to derive the behavior from StructureDefinitions without a special case.
