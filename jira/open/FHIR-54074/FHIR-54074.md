# FHIR-54074: %server is a long way from normative ready move to incubator

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/13/26
- **Description**: My general feedback for this is that "maybe" search/read, capabilities, and everything() could be reasonable, and maybe transform, but rest feels impractical for usage inside a fhirpath function evaluation.
The manipulation functions should need to be explicitly declared in a context to support those being included anywhere. Yay, someone just injected a trace with some side-effect resource creation in a search parameter, or an invariant, or a questionnaire, leaking sensitive data - quite the attack vector this could create. * Should not be normative, has not had suffient implementation experience - do ANY engines implement this?

	Was draft - maturity=0 in R5 (how many implementations/issues logged), inappropriate to go direct to normative
	Should have security implications included here
	A massive note should be included that functions that modify data should be used with great care, and very few implementations/contexts are likely to support them.
	how will error reporting be handled?
	how will authorization be covered?
	what problem is this trying to solve?
	what context is this expected to be use/available in?
Functions that modify/create content must be noted that these are ONLY available in specific contexts where they are clearly documented and not expected to be generally available.
	how is this related to x-fhir-query?
	search has the doPost as the first parameter? (should be optional and second, with default of false - to not do POST)
	the patch function has a different parameters to all the other functions here. How is the parameters object here expected to be created in a fhirpath expression?
	the transform function could just be a simple function transformStructure(canonicalUrl: string) - much simpler usage
	Apply should at least refer to the actual operation



    - doesn't get a resource from the server, isn't it creating a new resource? * validate? there is already a conformsTo function that can do this for profiles. And I'm confused why this would be a fhirpath function, and who would support it. Parameters parameter name inconsistent with others.

	Error reporting/handling here is terrible. just return nothing if a create failed?
	update doesn't mention anything about versioning stuff, or reference that part of the specification.
	no error handling/reporting for delete?
	deletions are done by id on the REST api, not the entire resource, should it have the same parameters as the read
	capabilities should also be handling the new feature capabilities stuff too
	capabilities should be returning the CapabilityStatement - not some resource
	everything doesn't "get a resource from the server", it's getting a set of resources from the server



I'm happy to go into more detail, I'd do that against the incubator to work on these. Or in the next ballot round (or just more trackers clarifying this) if desiring to retain this inside the normative spec.
