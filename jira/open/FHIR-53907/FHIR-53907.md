# FHIR-53907: SearchParameter.expression should define a FHIRPath subset you SHOULD use

## Metadata

- **Assignee**: Unassigned
- **Created**: 1/9/26
- **Description**: This is in response to this request:
Balloter feedback requested: We are seeking input on approaches to managing the balance between expressiveness and performance impact on servers from the use of various FHIRPath features in search parameter expressions.
In my opinion, it would be very helpful for servers if FHIR defined a limited subset of FHIRPath that one SHOULD use for SearchParameter.expression .

 

 

For my FHIR server implementation, we've implemented limited filtering capabilities based on SearchParameter FHIRPaths. For performance reasons, we do not implement a full FHIRPath engine, but rather only the subset needed for search parameters. This ends up requiring a very limited subset of FHIRPath, but we could only derive this subset by manually parsing/evaluating the SearchParameter definitions.

 

Indicating the subset of FHIRPath in use by the base spec would have made this easier.

More importantly, providing SHOULD guidance on what subset of FHIRPath to use would encourage future IGs to design reasonably computable custom search parameters.

Additionally, there's a few variations to communicate the same functional concept in FHIRPath, so SHOULD guidance would also help in standardizing that.  For example, compare Observation's {{value-date }}expression in STU3 vs R4 vs R6. This parameter has 3 different variations in syntax that are all functionally identical.

(in fact, even R6 is not fully self-consistent. Compare ActivityDefinition.subject-canonical with Task.focus-canonical )
